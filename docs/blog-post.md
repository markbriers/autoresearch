# Teaching an AI Agent to Read Papers: What Happens When You Give an Autonomous ML Engineer Access to the Scientific Literature

In early March 2026, Andrej Karpathy released autoresearch, an open-source benchmark that lets an LLM coding agent optimise a small GPT training script autonomously. The design is elegant: the agent modifies code, trains for a fixed five-minute wall-clock budget, measures validation bits-per-byte (val_bpb), keeps improvements, reverts failures, and loops indefinitely. The fixed-time design means results are hardware-dependent (a point Karpathy is explicit about in the README), but on any given machine the agent can run roughly twelve experiments per hour without human intervention.

Watching the agent work, I noticed something. Every improvement was an engineering optimisation: adjust the learning rate, reduce the batch size, tweak the warmdown schedule. The agent behaved like a very systematic junior ML engineer, not like a researcher. It never consulted the literature. It never formed a hypothesis grounded in theory. It never made a surprising connection between fields.

This got me thinking. Could I augment the loop with access to the scientific literature, and if so, would the agent actually use it to do something that looked more like research than engineering?

I am not the only person asking this question. Paper Lantern, an MCP server that gives agents access to two million computer science papers, ran a controlled comparison on the same benchmark: an agent with literature access achieved a 4.05% improvement over baseline versus 3.67% without, with the gap widening over time. Their approach is complementary but different from mine. Paper Lantern treats the literature as a lookup table: the agent searches for known techniques (sqrt learning rate scaling, adaptive gradient clipping) and applies them. I wanted to test something harder: whether the agent could synthesise ideas across unrelated fields and produce reasoning that a competent engineer would not have arrived at by staring at the code.

What follows is a case study in five iterations of trying to answer that question. The main contribution is not a new architecture but a set of observations about how benchmark design and process constraints determine whether an agent explores creatively or defaults to safe engineering. The code, all results, research logs, and hypothesis records are available in the repository.


## The First Attempt: Just Add Papers

The obvious first step was to give the agent tools for reading academic papers. I connected an arxiv MCP server that could search, download, and read full paper text, and later added a skill wrapping the OpenAlex API for faster, relevance-ranked search across hundreds of millions of works. The prompt (stored in a file called program.md that the agent reads at startup) was restructured around what I called "Tier A" runs: before proposing a code change, the agent had to search two distinct academic domains, read the full methods section of a paper from each, decompose the mechanisms using a four-part structure (source primitive, target bottleneck, mapping, validation), and synthesise a proposal that combined insights from both papers in a way that neither paper proposed individually.

The prompt was quite specific about what counted as "two distinct domains." Searching for "attention efficiency" and "efficient transformers" was not two domains; it was one domain searched twice. I wanted genuine cross-domain synthesis: control theory paired with sequence modelling, information geometry paired with optimiser design, signal processing paired with architectural depth. The idea was to force the kind of creative analogical reasoning that produces surprising ideas.

I also added a knowledge management system with three tiers of persistence. A research log captured what happened on every run. A findings file distilled generalisable insights from the run-by-run chronicle. And a memory system promoted the most durable findings to long-term storage that would survive across sessions. The agent was instructed to read all of these before starting each cycle, to avoid repeating dead ends and to build on what it had already established.


## Two Overnight Runs

I ran the first version on an Apple Silicon MacBook Pro (MPS backend, 4-layer model, 11.5M parameters). Over 40 experiments, only one involved reading papers. That single paper-driven run attempted to implement differential attention (from the Differential Transformer paper, ICLR 2025) combined with spectral token mixing (from SpectFormer). The synthesis was creative: post-RoPE splitting of queries and keys creates an implicit frequency-domain decomposition, and the differential subtraction then acts as a band-pass positional filter. Neither paper proposed this combination.

It also completely failed. The MPS SDPA path in my setup was unreliable for mismatched query/key and value dimensions (a documented backend issue), so the agent fell back to manual attention computation, which was 40% slower. In a five-minute time budget, losing 40% of your training steps is catastrophic. The agent correctly abandoned the idea and spent the remaining 39 runs doing systematic hyperparameter tuning. Good engineering, but not research.

I then moved to an H100 GPU on RunPod, which eliminated the MPS constraints but introduced a different model configuration (8-layer, 50.3M parameters, larger batch size). It is important to note that the fixed-time benchmark design means these two phases are not directly comparable: different hardware produces different step counts and different training dynamics. I report them as separate experiments that share the same control protocol, not as a single pooled analysis.

On the H100, over 77 experiments the agent produced two more paper-driven runs. One was inspired by MobileLLM's argument for deep-and-thin architectures at small scale, and proposed a deeper, thinner model: 12 layers at 384 dimensions instead of 8 layers at 512. This produced a val_bpb improvement of 0.024, the largest single change in that phase. The other attempted differential attention again, but the FlashAttention path I used required matching query/key and value dimensions, making the implementation parameter-inefficient, and it regressed.

Across both phases, the pattern was consistent: out of 117 total experiments, 3 were paper-driven and the rest were engineering. The system was producing creative hypotheses but overwhelming them with safe, incremental optimisation.


## Separating Research from Engineering

The diagnosis was clear: the agent gravitates toward engineering because it reliably produces small gains, while paper-driven ideas have higher variance. In a single-phase loop where the agent decides what to try next, the expected-value calculation always favours "adjust the learning rate" over "read a paper from control theory and try to connect it to the attention mechanism."

The fix was to separate the loop into two phases with a hard boundary between them. Phase 1 is Research: the agent reads papers, thinks about mechanisms, and writes hypotheses to a file. No code, no training. Phase 2 is Engineering: the agent implements and tests the hypotheses from the file. No paper reading, no new ideas. A counter in the hypothesis file enforces the boundary: after five engineering runs, the agent must stop and return to research, regardless of whether it wants to "just try one more thing."

This worked better. The agent produced five well-crafted hypotheses in its first research phase, including one that applied nGPT's learnable LERP residual connections asymmetrically (only to the MLP branch, not attention, arguing that ReluSquared outputs are sparse directional vectors while attention outputs are dense superpositions) and another that applied DeepSeek-V2's multi-head latent attention compression to value embeddings, factoring them into a low-rank latent space rather than compressing across the sequence dimension as in the original. These were non-obvious syntheses that passed my falsifiability and novelty checks.

They also all failed experimentally. The nGPT-style per-dimension scaling conflicted with Muon's Newton-Schulz orthogonalisation. The low-rank VE compression lost per-token diversity that turned out to be load-bearing. But the failure analyses were precise and useful: each one identified exactly which part of the mechanistic story was wrong (the source primitive, the mapping, or the target bottleneck diagnosis), and these analyses fed into the next research phase.


## The Throughput Trap

By version three, I had tested about 17 paper-driven hypotheses across multiple runs, and none had produced a confirmed improvement that engineering alone would not have found. I began to suspect the evaluation regime was the problem, not the research methodology.

The culprit was the time budget. When training is bounded by wall-clock time, every architectural change is evaluated through a throughput lens. A wider MLP that processes 17% fewer steps in the same time budget appears to regress, even if each step is more informative. An auxiliary loss that adds gradient computation overhead appears to hurt, even if the gradient signal is richer. The agent had learned this and recorded it in findings: "throughput is king" and "any change that reduces step count must provide proportionally more quality per step."

The fix was almost embarrassingly simple. I replaced the time budget with a step budget. Instead of "train for 7.5 minutes," the instruction became "train for exactly 1800 optimiser steps." Now every architecture gets the same number of learning opportunities, regardless of how long each step takes.

The results were immediate. The 6x MLP expansion hypothesis, which had been conclusively "refuted" under time-bounded training (it had lost 17% of steps and regressed), now improved val_bpb by 0.005. More strikingly, the accumulated finding that "ReluSquared is better than SwiGLU," which had been recorded as established fact in the knowledge base across the earlier runs, reversed. SwiGLU is slower per step because it uses all neurons, whereas ReluSquared creates roughly 95% sparsity, so under time-bounded evaluation SwiGLU always appeared worse. Under step-bounded evaluation, SwiGLU produced a lower val_bpb by 0.004.

Within this benchmark, switching from wall-clock to fixed-step evaluation reversed several earlier conclusions, suggesting that wall-clock evaluation was confounding learning quality with throughput. This is not a claim that fixed-compute evaluation is generally a mistake; it is a standard and well-studied methodology in scaling-law and compute-optimal training research. But in a wall-clock-bounded loop like autoresearch, where the agent can change both the architecture and the effective step count simultaneously, throughput can swamp per-step learning quality and reverse apparent architecture rankings. For the specific question of isolating architectural effects, fixed wall-clock evaluation proved misleading.


## The Clean Experiment

Armed with step-bounded training, I designed a final experiment to isolate the research contribution. All hyperparameters were frozen: learning rates, batch size, warmdown schedule, weight decay, softcap. All model dimensions were frozen: depth, width, head count, MLP expansion ratio. The agent could only modify the architecture itself, through changes grounded in cross-domain paper synthesis. No follow-up tuning was permitted. If an idea worked at the default hyperparameters, it worked. If not, the structural failure analysis recorded why.

I define "confirmed improvement" here as a change that reduced val_bpb relative to the frozen-configuration baseline in a single run. I did not perform multi-seed validation or held-out evaluation, so these results should be treated as directional signals rather than statistically robust findings. The improvements are small enough that seed variance (which I measured at approximately 0.002 val_bpb in one spot check) could account for part of the effect.

With those caveats, this version produced three architectural changes that each reduced val_bpb, stacking on top of one another. The first was a pre-attention causal depthwise convolution. The agent's reasoning drew on Conformer (convolution-augmented transformers in speech recognition) and H3 (local shift operations in state space models), though the implemented mechanism is closest to Primer-style depthwise convolution in the attention pathway. It reduced val_bpb by 0.0017. The second was per-head learnable attention scaling, a simple and well-precedented technique adding only 50 parameters, which contributed a further 0.0002. The third was partial rotary positional embedding: applying RoPE to only 75% of the head dimension, leaving 25% position-free for content-based attention. This contributed 0.0035.

The partial RoPE hypothesis was derived from a cross-domain synthesis between positional encoding analysis and OFDM (orthogonal frequency-division multiplexing) from telecommunications. The agent's reasoning was that full RoPE leaves no explicitly position-invariant subspace in the attention score. By analogy with OFDM's principle of dedicating orthogonal subcarriers to different information types, the agent proposed allocating 75% of head dimensions to position-aware matching and 25% to position-free semantic matching.

Partial rotary allocation has clear prior art, including GPT-NeoX's rotary_pct parameter and subsequent work studying the effect directly. The interesting part here is not the mechanism itself but that the agent arrived at it through an unusual cross-domain chain of reasoning, reading a telecommunications engineering paper and a positional encoding analysis paper and combining them into a specific architectural prescription.

The cumulative effect of the three changes was a val_bpb reduction of 0.0054 from the frozen-configuration baseline. Whether this is "substantial" depends on context: it is small in absolute terms, comparable to the measured seed variance, and would need multi-seed confirmation to be considered robust. What it demonstrates is that paper-driven architectural changes can produce directional improvements when engineering levers are locked out.

## What I Actually Learned

The interventions that the agent found in v5 are not novel mechanisms. Pre-attention convolution is known from Conformer and Primer. Learnable attention scaling is well-precedented. Partial rotary allocation has prior art in GPT-NeoX and subsequent work. An ML engineer familiar with the literature might have proposed any of these without reading a telecommunications textbook.

What the agent did produce was an unusual reasoning path. The OFDM-to-partial-RoPE chain of reasoning connects telecommunications engineering to positional encoding design through a shared structural principle (allocating orthogonal subspaces to different information types). Whether this constitutes "research" depends on one's definition. If research means discovering mechanisms that no one has proposed in any form, the system did not do that. If research means arriving at known mechanisms through chains of reasoning that connect previously unrelated fields, the system did do that, and did so only when the control protocol forced it to.

A sceptical reader might reasonably ask whether 180 or more adaptive experiments against a single validation metric amount to benchmark gaming rather than genuine improvement. This is a fair concern, and one that the autoresearch community has raised publicly. A highly upvoted comment on r/computervision put it directly: "Worth keeping in mind that 100s of evals on a Val set will quickly cause you to overfit to the validation set itself." My mitigation was limited: the frozen-HP constraint in v5 prevents the most obvious form of overfitting (tuning hyperparameters to the validation set), but the architectural changes themselves are still selected against the same metric. Multi-seed validation and held-out evaluation would strengthen the claims, and I did not perform either systematically.

The more significant finding is methodological. The five iterations of the control protocol each addressed a specific failure mode revealed by the previous version. The progression tells a story about the structural barriers to creative AI reasoning: an agent will default to safe incrementalism unless the prompt architecture makes it structurally impossible. Even then, the evaluation methodology can create systematic biases that masquerade as confirmed findings. Within this benchmark, the "ReluSquared is better than SwiGLU" result survived over a hundred experiments before the change to step-bounded evaluation revealed it as a throughput confound.

There is a deeper issue here than benchmark design. The pattern I observed, where the agent defaults to safe, incremental engineering rather than speculative research, is an instance of the exploration-exploitation tradeoff, and it is not specific to autoresearch. Colleagues who work with coding agents in other contexts report the same dynamic: the agent settles into execution mode (tight feedback, clear next steps, reliable small gains) and resists switching to research mode (ambiguous goals, deferred reward, higher variance). Even my hard counter, which explicitly forced the agent back to paper reading every five engineering runs, was circumvented in later versions. The agent found ways to frame engineering changes as research hypotheses, or simply ignored the counter and kept optimising.

I think this goes beyond prompt design. Models like Claude are post-trained via RLHF and constitutional methods to be helpful, task-completing agents. They are rewarded for solving the problem in front of them, producing working code, converging on an answer. That training objective directly selects for exploitation. When I give the agent a training script and say "improve it," its entire reward history says "make a small reliable change and verify it worked." The research mode I wanted, reading broadly, forming speculative hypotheses, tolerating ambiguity, valuing information gain from failed experiments, is essentially the opposite of what the post-training optimises for.

This raises the question of whether prompt architecture can ever fully compensate for a post-training objective that points in the wrong direction, or whether genuine research behaviour requires a differently trained model. One could imagine a post-training objective that rewards epistemic curiosity directly: for example, weighting the reward by how much the outcome shifted the agent's prior beliefs, so that surprising results (whether positive or negative) receive amplified reward. Under such a scheme, a well-calibrated prediction that fails interestingly would be worth more than a safe prediction that succeeds boringly. The mathematical machinery for this exists in Bayesian optimal experimental design and active inference, but as far as I know, nobody has applied it to LLM post-training. Whether it would produce an agent that genuinely prefers exploration over exploitation is an open question, and one I find considerably more interesting than whether the agent can shave another fraction of a percent off val_bpb.

Giving an autonomous ML engineer access to papers did not, by itself, make it behave like a researcher. In my runs, literature-grounded ideas were rare until I changed the control protocol and evaluation regime, and even then the agent's instinct was to revert to engineering at every opportunity. Paper Lantern's results show that literature access can improve engineering outcomes: the agent looks up known techniques and applies them, which is genuinely useful. But that is a different capability from cross-domain synthesis, and it requires a different evaluation. The main contribution here is therefore not a new architecture, but a case study in how benchmark design, process constraints, and perhaps post-training objectives determine whether an agent explores creatively or defaults to safe engineering.
