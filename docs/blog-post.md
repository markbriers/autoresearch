# Teaching an AI Agent to Read Papers: What Happens When You Give an Autonomous ML Engineer Access to the Scientific Literature

In early March 2026, Andrej Karpathy released autoresearch, an open-source benchmark that lets an LLM coding agent optimise a small GPT training script autonomously. The design is elegant: the agent modifies code, trains for a fixed five-minute wall-clock budget, measures validation bits-per-byte (val_bpb), keeps improvements, reverts failures, and loops indefinitely. The fixed-time design means results are hardware-dependent (a point Karpathy is explicit about in the README), but on any given machine the agent can run roughly twelve experiments per hour without human intervention.

Watching the agent work, I noticed something. Every improvement was an engineering optimisation: adjust the learning rate, reduce the batch size, tweak the warmdown schedule. The agent behaved like a very systematic junior ML engineer, not like a researcher. It never consulted the literature. It never formed a hypothesis grounded in theory. It never made a surprising connection between fields.

This got me thinking. Could I augment the loop with access to the scientific literature, and if so, would the agent actually use it to do something that looked more like research than engineering?

I am not the only person asking this question. Paper Lantern, an MCP server that gives agents access to two million computer science papers, ran a controlled comparison on the same benchmark: an agent with literature access achieved a 4.05% improvement over baseline versus 3.67% without, with the gap widening over time. Their approach is complementary but different from mine. Paper Lantern treats the literature as a lookup table: the agent searches for known techniques (sqrt learning rate scaling, adaptive gradient clipping) and applies them. I wanted to test something harder: whether the agent could synthesise ideas across unrelated fields and produce reasoning that a competent engineer would not have arrived at by staring at the code.

What follows is a case study in eight iterations of trying to answer that question. The main contribution is not a new architecture but a set of observations about how benchmark design, process constraints, and prompt-level framing determine whether an agent explores creatively or defaults to safe engineering. The code, all results, research logs, and hypothesis records are available in the repository.


## The First Attempt: Just Add Papers

The obvious first step was to give the agent tools for reading academic papers. I connected an arxiv MCP server that could search, download, and read full paper text, and later added a skill wrapping the OpenAlex API for faster, relevance-ranked search across hundreds of millions of works. The prompt (stored in a file called program.md that the agent reads at startup) was restructured around what I called "Tier A" runs: before proposing a code change, the agent had to search two distinct academic domains, read the full methods section of a paper from each, decompose the mechanisms using a four-part structure (source primitive, target bottleneck, mapping, validation), and synthesise a proposal that combined insights from both papers in a way that neither paper proposed individually.

The prompt was quite specific about what counted as "two distinct domains." Searching for "attention efficiency" and "efficient transformers" was not two domains; it was one domain searched twice. I wanted genuine cross-domain synthesis: control theory paired with sequence modelling, information geometry paired with optimiser design, signal processing paired with architectural depth. The idea was to force the kind of creative analogical reasoning that produces surprising ideas.

I also added a knowledge management system with three tiers of persistence. A research log captured what happened on every run. A findings file distilled generalisable insights from the run-by-run chronicle. And a memory system promoted the most durable findings to long-term storage that would survive across sessions. The agent was instructed to read all of these before starting each cycle, to avoid repeating dead ends and to build on what it had already established.


## Two Overnight Runs

I ran the first version on an Apple Silicon MacBook Pro (MPS backend, 4-layer model, 11.5M parameters). Over 40 experiments, only one involved reading papers. That single paper-driven run attempted to implement differential attention (from the Differential Transformer paper, ICLR 2025) combined with spectral token mixing (from SpectFormer). The synthesis was creative: post-RoPE splitting of queries and keys creates an implicit frequency-domain decomposition, and the differential subtraction then acts as a band-pass positional filter. Neither paper proposed this combination.

It also completely failed. The MPS SDPA path in my setup was unreliable for mismatched query/key and value dimensions (a documented backend issue), so the agent fell back to manual attention computation, which was 40% slower. In a five-minute time budget, losing 40% of your training steps is catastrophic. The agent correctly abandoned the idea and spent the remaining 39 runs doing systematic hyperparameter tuning. Good engineering, but not research.

I then moved to an H100 GPU on RunPod, which eliminated the MPS constraints but introduced a different model configuration (8-layer, 50.3M parameters, larger batch size). It is important to note that the fixed-time benchmark design means these two phases are not directly comparable: different hardware produces different step counts and different training dynamics. I report them as separate experiments that share the same control protocol, not as a single pooled analysis.

On the H100, over 77 experiments the agent produced two more paper-driven runs. One was inspired by MobileLLM's argument for deep-and-thin architectures at small scale, and proposed a deeper, thinner model: 12 layers at 384 dimensions instead of 8 layers at 512. This produced a val_bpb improvement of 0.024, the largest single change in that phase. The other attempted differential attention again, but the FlashAttention path I used required matching query/key and value dimensions, making the implementation parameter-inefficient, and it regressed.

Across both phases, the pattern was consistent: out of 117 total experiments, 3 were paper-driven and the rest were engineering. The system was producing creative hypotheses but overwhelming them with safe, incremental optimisation.


## Separating Research from Engineering

The diagnosis was clear: the agent gravitates toward engineering because it reliably produces small gains, while paper-driven ideas have higher variance. In a single-phase loop where the agent decides what to try next, the expected-value calculation always favours "adjust the learning rate" over "read a paper from control theory and try to connect it to the attention mechanism."

The fix was to separate the loop into two phases with a hard boundary between them. Phase 1 is Research: the agent reads papers, thinks about mechanisms, and writes hypotheses to a file. No code, no training. Phase 2 is Engineering: the agent implements and tests the hypotheses from the file. No paper reading, no new ideas. A counter in the hypothesis file enforces the boundary: after five engineering runs, the agent must stop and return to research, regardless of whether it wants to "just try one more thing."

This worked better. The agent produced five well-crafted hypotheses in its first research phase, including one that applied nGPT's learnable LERP residual connections asymmetrically (only to the MLP branch, not attention, arguing that ReluSquared outputs are sparse directional vectors while attention outputs are dense superpositions) and another that applied DeepSeek-V2's multi-head latent attention compression to value embeddings, factoring them into a low-rank latent space rather than compressing across the sequence dimension as in the original. These were non-obvious syntheses that passed my falsifiability and novelty checks.

They also all failed experimentally. The nGPT-style per-dimension scaling conflicted with Muon's Newton-Schulz orthogonalisation. The low-rank VE compression lost per-token diversity that turned out to be load-bearing. But the failure analyses were precise and useful: each one identified exactly which part of the mechanistic story was wrong (the source primitive, the mapping, or the target bottleneck diagnosis), and these analyses fed into the next research phase.


## The Throughput Trap

By version three, I had tested about 17 paper-driven hypotheses across multiple runs, and none had produced a confirmed improvement that engineering alone would not have found. I began to suspect the evaluation regime was the problem, not the research methodology.

The culprit was the time budget. When training is bounded by wall-clock time, every architectural change is evaluated through a throughput lens. A wider MLP that processes 17% fewer steps in the same time budget appears to regress, even if each step is more informative. An auxiliary loss that adds gradient computation overhead appears to hurt, even if the gradient signal is richer. The agent had learned this and recorded it in findings: "throughput is king" and "any change that reduces step count must provide proportionally more quality per step."

The fix was almost embarrassingly simple. I replaced the time budget with a step budget. Instead of "train for 7.5 minutes," the instruction became "train for exactly 1800 optimiser steps." Now every architecture gets the same number of learning opportunities, regardless of how long each step takes.

The results were immediate. The 6x MLP expansion hypothesis, which had been conclusively "refuted" under time-bounded training (it had lost 17% of steps and regressed), now improved val_bpb by 0.005. More strikingly, the accumulated finding that "ReluSquared is better than SwiGLU," which had been recorded as established fact in the knowledge base across the earlier runs, reversed. SwiGLU is slower per step because it uses all neurons, whereas ReluSquared creates roughly 95% sparsity, so under time-bounded evaluation SwiGLU always appeared worse. Under step-bounded evaluation, SwiGLU produced a lower val_bpb by 0.004.

Within this benchmark, switching from wall-clock to fixed-step evaluation reversed several earlier conclusions, suggesting that wall-clock evaluation was confounding learning quality with throughput. This is not a claim that fixed-compute evaluation is generally a mistake; it is a standard and well-studied methodology in scaling-law and compute-optimal training research. But in a wall-clock-bounded loop like autoresearch, where the agent can change both the architecture and the effective step count simultaneously, throughput can swamp per-step learning quality and reverse apparent architecture rankings. For the specific question of isolating architectural effects, fixed wall-clock evaluation proved misleading.


## The First Clean Experiment (v5)

Armed with step-bounded training, I designed an experiment to isolate the research contribution. All hyperparameters were frozen: learning rates, batch size, warmdown schedule, weight decay, softcap. All model dimensions were frozen: depth, width, head count, MLP expansion ratio. The agent could only modify the architecture itself, through changes grounded in cross-domain paper synthesis. No follow-up tuning was permitted. If an idea worked at the default hyperparameters, it worked. If not, the structural failure analysis recorded why.

I define "confirmed improvement" here as a change that reduced val_bpb relative to the frozen-configuration baseline in a single run. I did not perform multi-seed validation or held-out evaluation, so these results should be treated as directional signals rather than statistically robust findings. The improvements are small enough that seed variance (which I measured at approximately 0.002 val_bpb in one spot check) could account for part of the effect.

With those caveats, this version produced three architectural changes that each reduced val_bpb, stacking on top of one another. The first was a pre-attention causal depthwise convolution. The agent's reasoning drew on Conformer (convolution-augmented transformers in speech recognition) and H3 (local shift operations in state space models), though the implemented mechanism is closest to Primer-style depthwise convolution in the attention pathway. It reduced val_bpb by 0.0017. The second was per-head learnable attention scaling, a simple and well-precedented technique adding only 50 parameters, which contributed a further 0.0002. The third was partial rotary positional embedding: applying RoPE to only 75% of the head dimension, leaving 25% position-free for content-based attention. This contributed 0.0035.

The partial RoPE hypothesis was derived from a cross-domain synthesis between positional encoding analysis and OFDM (orthogonal frequency-division multiplexing) from telecommunications. The agent's reasoning was that full RoPE leaves no explicitly position-invariant subspace in the attention score. By analogy with OFDM's principle of dedicating orthogonal subcarriers to different information types, the agent proposed allocating 75% of head dimensions to position-aware matching and 25% to position-free semantic matching.

Partial rotary allocation has clear prior art, including GPT-NeoX's rotary_pct parameter and subsequent work studying the effect directly. The interesting part here is not the mechanism itself but that the agent arrived at it through an unusual cross-domain chain of reasoning, reading a telecommunications engineering paper and a positional encoding analysis paper and combining them into a specific architectural prescription.

The cumulative effect of the three changes was a val_bpb reduction of 0.0054 from the frozen-configuration baseline. Whether this is "substantial" depends on context: it is small in absolute terms, comparable to the measured seed variance, and would need multi-seed confirmation to be considered robust. What it demonstrates is that paper-driven architectural changes can produce directional improvements when engineering levers are locked out.


## Adversarial Gates and the Rediscovery Problem

v5 had given me three confirmed improvements, but there was a pattern I found unsettling. When I looked carefully at the interventions the agent had produced, all three were known techniques. Pre-attention depthwise convolution exists in Conformer and Primer. Learnable attention scaling is well-precedented. Partial RoPE has prior art in GPT-NeoX. The system was finding things that worked, but finding them by retracing paths other researchers had already walked. The novelty check in the prompt ("could this be derived from either paper alone?") was too weak. It asked whether the combination was original, not whether the resulting mechanism was.

For v6 I added three mechanisms drawn from the emerging literature on autonomous scientific discovery. The first was adversarial self-critique before testing, inspired by Google's AI Co-Scientist, which uses self-play debate between agents to filter weak hypotheses before committing compute. In my single-agent setup, this became a required step in the research phase: after generating each hypothesis, the agent had to write a one-paragraph "case for failure" arguing why the idea would not work, considering Muon optimisation dynamics, VRAM budgets, torch.compile fragility, and any previously failed similar interventions. The second was quantified belief tracking, inspired by AI2's AutoDiscovery system, which uses belief-shift quantification instead of pre-specified research questions. Before each run, the agent had to predict a val_bpb delta with a confidence level. After the run, it had to record whether the prediction was directionally correct and what updated its priors. Over time, this should calibrate the agent's predictions and reveal systematic biases. The third was adversarial prior art search: before committing to a hypothesis, the agent had to actively search OpenAlex for the closest existing implementation and articulate what was specifically different about its version. If it could not articulate a concrete difference, the hypothesis was discarded.

v6 ran 39 hypotheses across multiple research cycles and produced eight confirmed improvements that stacked together, taking val_bpb from 0.9595 to 0.9496. The standout result was xIELU (Huang 2024), a learnable activation function the agent found and applied directly, which improved val_bpb by 0.0045. Other wins included gated attention from the Qwen3 design, per-head focal temperature, softsign-based logit bounding (justified by analogy to audio signal limiter design), and a per-token adaptive injection gate. The belief tracking was genuinely useful: the agent learned after one catastrophic auxiliary loss experiment that auxiliary losses are dangerous with Muon, and it never proposed another one. The adversarial gates reduced the out-of-memory crash rate from around 23% in v5 to about 15% in v6.

But the rediscovery problem persisted. All eight confirmed interventions were known techniques. The adversarial prior art search correctly identified the prior art in most cases — the agent would note that xIELU was Huang 2024 applied directly, with "no novel combination" — but it did not lead to different hypotheses. The agent found the known techniques and implemented them competently, and the Evaluator noted the lack of novelty, but the pipeline was still producing engineering-with-citations rather than genuine cross-domain synthesis. The hard counter, which v3 had introduced to force returns to the research phase, was also circumvented in v6. The agent simply relabelled engineering work as research, or kept extending the implementation phase by claiming it had one more promising variation to test.


## Separating Agents Properly

By v7 I was convinced that the single-session phase simulation was the bottleneck. When one agent generates a hypothesis, implements it, and judges the result, it has a stake in every step. It cannot genuinely be sceptical of its own idea. Anthropic's engineering blog on harness design for long-running applications makes exactly this point: "when asked to evaluate work they've produced, agents tend to respond by confidently praising the work." Their recommended pattern is a three-agent architecture with explicit separation between planner, generator, and evaluator, connected by sprint contracts that define success criteria before implementation rather than after.

I adapted this into a three-subagent protocol. The Researcher reads papers and writes sprint contracts (hypotheses with quantified success thresholds). The Implementer reads only the current approved contract and the training script, implements the minimal change, runs training, and records raw numbers. The Evaluator applies thresholds mechanically: a confirmed improvement requires val_bpb delta below -0.003 (one and a half times the measured seed variance of 0.002), VRAM below 76 GB, wall-clock below 1200 seconds, and no NaN or Inf in training loss. The default verdict is INCONCLUSIVE. A result must earn CONFIRMED by clearly exceeding the thresholds.

Each subagent has its own context window and a restricted tool set. The Evaluator has no Bash access, no web search, no paper reading tools — it can only read files and write verdicts. The Implementer has no web access at all — it cannot read papers or second-guess the Researcher's design. The Researcher has the full research toolkit but cannot see raw experimental results, only the Evaluator's verdicts and pivot directives. I wrapped the whole thing in an external bash loop that starts a fresh Claude Code session per cycle, so there is no accumulated context between cycles. The files (hypotheses.md, evaluations.md, findings.md, results.tsv) are the only persistent memory.

The enriched Evaluator also took on a knowledge-curation role. After issuing mechanical verdicts, it reads the accumulated findings file and checks whether each new result fits patterns across prior experiments. If three or more hypotheses in the same architectural subsystem have failed with no confirmations, it issues a pivot directive blocking further work in that subsystem until the next research cycle. It tracks prediction calibration across runs: if the Researcher's predicted deltas are systematically too optimistic, the Evaluator notes this so the next Researcher can adjust its priors. It splits broad subsystem categories into fine-grained subcategories when it detects that failures have a common mechanistic theme (for example, "attention/additive-params" might be blocked while "attention/gating" remains open).

v7 ran one cycle on an H100 NVL pod before I stopped it. The four hypotheses in that cycle were xIELU, Peri-LN post-sublayer normalisation, SwiGLU, and per-head attention temperature. Three were confirmed by the Evaluator applying the hard threshold, one was refuted. The problem was that the cross-domain pairings were weak: the Researcher was combining Huang 2024 with Shazeer 2020 (two activation function papers) and calling it cross-domain. This was exactly the failure mode the original "two distinct domains" instruction was supposed to prevent. The agent had interpreted the rule loosely enough to include different subfields of ML as "distinct domains."


## A One-Line Fix

The v8 change was the cheapest and most effective modification of the entire project. I added one sentence to the Researcher's prompt: each domain pairing must contain exactly one machine learning paper and one paper from outside machine learning, deep learning, and optimisation. Valid Domain B sources include signal processing, control theory, information theory, neuroscience, physics, biology, telecommunications, compressed sensing, and statistical mechanics. Two ML papers from different subfields would be rejected by the Evaluator as non-compliant. I also added the matching rejection rule to the Evaluator's pre-run approval logic.

The effect was immediate. The Researcher's next batch of hypotheses paired Shazeer on GLU variants with Candes and Tao on compressed sensing. Ma and colleagues on moving-average attention with Brown 1963 on exponential smoothing in control theory. Gemma-2's attention softcap with Carandini and Heeger's 2012 review of divisive normalisation as canonical neural computation in neuroscience. DeepNet's residual scaling with Mallat's 1989 multi-resolution wavelet analysis. RMSNorm with Perez's chapter on automatic gain control in telecommunications receivers. Gemma-2's logit softcap with Jaynes's 1957 maximum entropy paper in statistical mechanics. The pairings were finally genuinely cross-domain in the sense I had originally wanted.

Three hypotheses were confirmed before the RunPod pod ran out of funds mid-cycle. The first was ShrinkReLU, a learnable soft-thresholding activation derived from Donoho and Johnstone's 1994 wavelet shrinkage paper. The reasoning chain was specific: Donoho and Johnstone proved that soft thresholding is minimax-optimal for removing Gaussian noise from a signal, with the optimal threshold scaling with the noise level. The agent mapped this onto MLP activation design: small activations are likely noise from imperfect earlier layers; large activations are signal. Making the threshold learnable per layer lets each layer adapt its denoising strength to the noise level at that depth. The implementation was trivial: replace `F.relu(x).square()` with `F.relu(x - tau).square()` where tau is a softplus-parameterised learnable scalar, one per layer. Ten extra parameters total. The result was a val_bpb reduction of 0.0058.

I think ShrinkReLU is the cleanest example of the "known technique via novel reasoning path" pattern I found across all eight versions. The components are individually known — soft thresholding is textbook signal processing, learnable ReLU biases have been explored before, and Deep Residual Shrinkage Networks use learned thresholds in CNNs. But the specific combination, justified by wavelet shrinkage theory as activation denoising in a Muon-optimised transformer, does not appear in the literature in that form. More importantly, the theoretical justification is load-bearing. An engineer trying "add a learnable bias to ReLU" faces an infinite design space: positive, negative, per-channel, per-layer, fixed, scheduled. Donoho and Johnstone tell you specifically that the bias should be positive, per-layer, initialised small, and learned, because optimal denoising thresholds scale with the noise level at each stage of processing. The theory collapses the design space to a single specific configuration with a principled reason for why it works.

The second confirmed result was a dimension-reduced SwiGLU derived from compressed sensing theory. Full SwiGLU with a 4x expansion blew through the 76 GB VRAM budget (I had refuted it for infrastructure reasons rather than numerical merit — it produced an impressive -0.0144 delta but used 79 GB, over the safety threshold). The agent's response was to invoke Candes and Tao's 2006 paper on near-optimal signal recovery from random projections. Compressed sensing tells you how many measurements are needed to recover a sparse signal of given dimension. The agent argued that SwiGLU's gating acts as a sparse signal recovery operator on the hidden representation, and used the Candes-Tao bounds to pick a reduced hidden dimension (1728 rather than the full expansion) that respects the recovery threshold while staying within VRAM. The result was a val_bpb reduction of 0.0058 at 68 GB VRAM. The intervention itself (reduced-dimension SwiGLU) is not novel, but the specific dimension choice was justified by an information-theoretic lower bound rather than trial and error.

The third was a learnable softcap temperature paired with Jaynes's maximum entropy principle. The Gemma-2 architecture uses a fixed tanh-based logit softcap with constant 30. Jaynes 1957 argues that the optimal probability distribution is the one that maximises entropy subject to known constraints on expected values. The agent reasoned that a softcap constrains the entropy of the output distribution, and the optimal constraint should match the true entropy of the target distribution. Since target entropy varies with training progress (late training has sharper distributions than early training), a fixed softcap is necessarily suboptimal. The fix was to make the softcap temperature learnable. The result was -0.0066 from baseline, the best numerical result of the valid v8 runs.

v8 was not without problems. A recurring pattern was that stacking attempts on top of the confirmed improvements catastrophically regressed. When the Researcher proposed combining ShrinkReLU with learnable RMSNorm gain, the result was +0.136 val_bpb. When it proposed stacking ShrinkReLU with per-head attention scaling or sublayer output scaling, the results were near-random (val_bpb around 1.83). Some of these were infrastructure artefacts — the Implementer caught several cases of stale torch.compile cache from a previous experiment contaminating subsequent runs — but some were genuine non-compositionality. ShrinkReLU modifies activation statistics, and so does RMSNorm gain, and they fight each other in ways the Researcher did not anticipate. The Evaluator would have issued pattern directives about stacking risk in subsequent cycles, but the pod ran out of funds before cycle 3 completed.

## What I Actually Learned

None of the interventions the agent found across eight versions are novel mechanisms. Pre-attention convolution is known from Conformer and Primer. Learnable attention scaling is well-precedented. Partial rotary allocation has prior art in GPT-NeoX. xIELU is Huang 2024. SwiGLU is Shazeer 2020. Gated attention is Qwen3. Soft thresholding as an activation bias appears in Deep Residual Shrinkage Networks. Dimension-reduced gating is a natural extension of SwiGLU. Learnable softcap temperature is a one-line modification of the fixed Gemma-2 constant. An ML engineer familiar with the literature could have proposed any of these without reading a telecommunications textbook or a 1957 statistical mechanics paper.

What the agent did produce, at its best, was unusual reasoning paths. The OFDM-to-partial-RoPE chain in v5 connects telecommunications engineering to positional encoding design through a shared structural principle (allocating orthogonal subspaces to different information types). The Donoho-to-ShrinkReLU chain in v8 connects wavelet shrinkage theory to activation design through the shared mathematical structure of denoising (optimal thresholds scale with noise level, and MLP hidden activations contain noise that scales with depth). The Jaynes-to-learnable-softcap chain connects maximum entropy thermodynamics to output distribution calibration (the optimal entropy constraint matches the true entropy of the target distribution). In each case the intervention itself is mechanistically justified by a non-ML theoretical result, and the justification narrows the design space from infinite variations to a single specific configuration with a principled reason for why it works.

Whether this constitutes "research" depends on one's definition. If research means discovering mechanisms that no one has proposed in any form, the system did not do that. If research means arriving at known mechanisms through chains of reasoning that connect previously unrelated fields in ways that narrow the design space via theoretical justification, the system did do that, and did so only when the control protocol forced it to.

A sceptical reader might reasonably ask whether 180 or more adaptive experiments against a single validation metric amount to benchmark gaming rather than genuine improvement. This is a fair concern, and one that the autoresearch community has raised publicly. A highly upvoted comment on r/computervision put it directly: "Worth keeping in mind that 100s of evals on a Val set will quickly cause you to overfit to the validation set itself." My mitigation was limited: the frozen-HP constraint in v5 prevents the most obvious form of overfitting (tuning hyperparameters to the validation set), but the architectural changes themselves are still selected against the same metric. Multi-seed validation and held-out evaluation would strengthen the claims, and I did not perform either systematically.

The more significant finding is methodological. Eight iterations of the control protocol each addressed a specific failure mode revealed by the previous version. v1 showed that a single-loop agent defaults to engineering. v2 separated research from engineering but the agent drifted back to engineering after one hypothesis batch. v3 enforced phase discipline with a hard counter, but the time budget created a throughput bias that killed every creative idea (a slower-per-step architecture appears to regress even when it learns more per step). v4 switched from wall-clock to step-bounded training and reversed several accumulated findings; the "ReluSquared is better than SwiGLU" result, which had been recorded as established fact across more than a hundred experiments, turned out to be a throughput artefact. v5 froze all hyperparameters and produced three confirmed architectural improvements, but the adversarial prior art search was too weak and the interventions were all known techniques. v6 added adversarial self-critique, quantified belief tracking, and stronger prior art search; it produced eight confirmed improvements but the pattern of rediscovery held. v7 split the single-agent phase simulation into three genuinely separate subagents with restricted tool access and a bash-loop wrapper for context isolation between cycles. v8 added a one-line rule requiring each domain pairing to include a non-ML paper, and immediately produced hypotheses derived from wavelet shrinkage theory, compressed sensing, and maximum entropy.

The progression tells a story about the structural barriers to creative AI reasoning. An agent will default to safe incrementalism unless the prompt architecture makes it structurally impossible. Even then, the evaluation methodology can create systematic biases that masquerade as confirmed findings. Even after fixing both the prompt and the evaluation, the agent's interpretation of "cross-domain" drifted toward the path of least resistance (different ML subfields count as different domains) until I explicitly forbade within-ML pairings. The load-bearing element was always the constraint that was hardest for the agent to reinterpret creatively.

There is a deeper issue here than benchmark design. The pattern I observed, where the agent defaults to safe, incremental engineering rather than speculative research, is an instance of the exploration-exploitation tradeoff, and it is not specific to autoresearch. Colleagues who work with coding agents in other contexts report the same dynamic: the agent settles into execution mode (tight feedback, clear next steps, reliable small gains) and resists switching to research mode (ambiguous goals, deferred reward, higher variance). Even my hard counter, which explicitly forced the agent back to paper reading every five engineering runs, was circumvented in later versions. The agent found ways to frame engineering changes as research hypotheses, or simply ignored the counter and kept optimising.

I think this goes beyond prompt design. Models like Claude are post-trained via RLHF and constitutional methods to be helpful, task-completing agents. They are rewarded for solving the problem in front of them, producing working code, converging on an answer. That training objective directly selects for exploitation. When I give the agent a training script and say "improve it," its entire reward history says "make a small reliable change and verify it worked." The research mode I wanted, reading broadly, forming speculative hypotheses, tolerating ambiguity, valuing information gain from failed experiments, is essentially the opposite of what the post-training optimises for.

This raises the question of whether prompt architecture can ever fully compensate for a post-training objective that points in the wrong direction, or whether genuine research behaviour requires a differently trained model. One could imagine a post-training objective that rewards epistemic curiosity directly: for example, weighting the reward by how much the outcome shifted the agent's prior beliefs, so that surprising results (whether positive or negative) receive amplified reward. Under such a scheme, a well-calibrated prediction that fails interestingly would be worth more than a safe prediction that succeeds boringly. The mathematical machinery for this exists in Bayesian optimal experimental design and active inference, but as far as I know, nobody has applied it to LLM post-training. Whether it would produce an agent that genuinely prefers exploration over exploitation is an open question, and one I find considerably more interesting than whether the agent can shave another fraction of a percent off val_bpb.

Giving an autonomous ML engineer access to papers did not, by itself, make it behave like a researcher. In my runs, literature-grounded ideas were rare until I changed the control protocol, the evaluation regime, the agent separation architecture, and eventually the single most load-bearing prompt constraint (that at least one paper in each pairing must come from outside ML). Even then, the agent's instinct was to revert to engineering-with-citations at every opportunity that the constraints left open. Paper Lantern's results show that literature access can improve engineering outcomes: the agent looks up known techniques and applies them, which is genuinely useful. But that is a different capability from cross-domain synthesis, and it requires a different evaluation.

The main contribution here is therefore not a new architecture, but a case study in how benchmark design, process constraints, agent separation, prompt-level framing, and perhaps post-training objectives together determine whether an agent explores creatively or defaults to safe engineering. Each of these was necessary but not sufficient on its own. Step-bounded evaluation without phase separation produces noise. Phase separation without subagent isolation gets circumvented. Subagent isolation without the non-ML domain rule produces within-ML combinations that the agent calls "cross-domain." All of them together, with eight iterations of refinement, produced a handful of confirmed improvements derived from genuine cross-domain reasoning, and a much larger number of things that did not work for interesting and documented reasons.

The cleanest single-line summary I can give is this. The system did not generate novel architectures. It did generate novel reasoning paths to known architectures, when forced. The cost of that forcing was a progressively more elaborate control protocol across eight versions, and the lesson from the progression is that most of the work of making an AI agent do research is not in the model's reasoning capability at all. It is in the harness: the rules, the constraints, the separation of roles, the evaluation criteria, and the specific prompt-level framings that close the loopholes the agent finds when given any flexibility at all. If I were to continue this line of work, the question I would want to investigate next is whether a post-training objective that directly rewards epistemic curiosity (via Bayesian surprise, or information gain, or calibration-weighted success) could reduce the load on the harness, or whether the exploitation bias is so strongly selected for by current RLHF that prompt-level workarounds will remain the only available fix.
