# Teaching an AI Agent to Read Papers: What Happens When You Give an Autonomous ML Engineer Access to the Scientific Literature

In early March 2026, Andrej Karpathy released autoresearch, a beautifully simple benchmark that lets an LLM coding agent optimise a small GPT training script autonomously. The agent modifies code, trains for five minutes, measures validation bits-per-byte, keeps improvements, reverts failures, and loops indefinitely. Leave it running overnight and you wake up to a hundred experiments, all completed while you slept. Within two weeks it had 41,000 GitHub stars and Shopify's CEO was running it on internal data.

The results were impressive. The agent found genuine improvements, typically 11-19% better than the baseline. But watching it work, I noticed something. Every improvement was an engineering optimisation: adjust the learning rate, reduce the batch size, tweak the warmdown schedule. The agent was behaving like a very systematic junior ML engineer, not like a researcher. It never once consulted the literature. It never formed a hypothesis grounded in theory. It never made a surprising connection between fields.

This got me thinking. Could we augment the loop with access to the scientific literature, and if so, would the agent actually use it to do something that looked more like research than engineering?

## The First Attempt: Just Add Papers

The obvious first step was to give the agent tools for reading academic papers. We connected an arxiv MCP server that could search, download, and read full paper text, and later added a skill wrapping the OpenAlex API for faster, relevance-ranked search across 200 million papers. The prompt (stored in a file called program.md that the agent reads at startup) was restructured around what we called "Tier A" runs: before proposing a code change, the agent must search two distinct academic domains, read the full methods section of a paper from each, decompose the mechanisms using a four-part structure (source primitive, target bottleneck, mapping, validation), and synthesise a proposal that combines insights from both papers in a way that neither paper proposed individually.

The prompt was quite specific about what counted as "two distinct domains." Searching for "attention efficiency" and "efficient transformers" was not two domains; it was one domain searched twice. We wanted genuine cross-domain synthesis: control theory paired with sequence modelling, information geometry paired with optimiser design, signal processing paired with architectural depth. The idea was to force the kind of creative analogical reasoning that produces surprising ideas.

We also added a knowledge management system with three tiers of persistence. A research log captured what happened on every run. A findings file distilled generalisable insights from the run-by-run chronicle. And a memory system promoted the most durable findings to long-term storage that would survive across sessions. The agent was instructed to read all of these before starting each cycle, to avoid repeating dead ends and to build on what it had already established.

## The Overnight Disappointment

We left the first version running overnight on a MacBook Pro. It completed 40 experiments. The results were deflating. Out of 40 runs, only one involved reading papers. That single paper-driven run attempted to implement differential attention (from the Differential Transformer paper at ICLR 2025) combined with spectral token mixing (from SpectFormer). The synthesis was genuinely creative: post-RoPE splitting of queries and keys creates an implicit frequency-domain decomposition, and the differential subtraction then acts as a band-pass positional filter. Neither paper proposed this combination. It was exactly the kind of cross-domain synthesis the prompt was designed to elicit.

It also completely failed. The MPS backend on Apple Silicon does not support SDPA with mismatched query/key and value dimensions, so the agent had to fall back to manual attention computation, which was 40% slower. In a five-minute time budget, losing 40% of your training steps is catastrophic. The agent correctly abandoned the idea and spent the remaining 39 runs doing what Karpathy's original system does: systematic hyperparameter tuning. It found that reducing batch size from 32 to 8 gave the biggest improvement (more optimiser steps in the fixed time budget), and that various learning rate and schedule adjustments helped at the margins. Good engineering, but not research.

We then moved to an H100 GPU on RunPod, which eliminated the MPS constraints. Over 77 more experiments, the agent produced two more paper-driven runs. One combined insights from the MobileLLM paper (deep-and-thin architectures for mobile) with scaling law analyses (depth scales more efficiently than width at small training budgets) to propose a deeper, thinner model: 12 layers at 384 dimensions instead of 8 layers at 512. This was the biggest single improvement of the entire run, reducing val_bpb by 0.024. The other paper-driven run attempted differential attention again, but the FA3 dimension-matching constraint made the implementation parameter-inefficient, and it regressed.

So after 117 experiments across two hardware platforms, the score was: 3 paper-driven runs out of 117, with 1 genuine improvement. The system was producing creative hypotheses but overwhelming them with safe, incremental engineering.

## Separating Research from Engineering

The diagnosis was clear: the agent gravitates toward engineering because it reliably produces small gains, while paper-driven ideas have higher variance. In a single-phase loop where the agent decides what to try next, the expected-value calculation always favours "adjust the learning rate" over "read a paper from control theory and try to connect it to the attention mechanism."

The fix was to separate the loop into two phases with a hard boundary between them. Phase 1 is Research: the agent reads papers, thinks about mechanisms, and writes hypotheses to a file. No code, no training. Phase 2 is Engineering: the agent implements and tests the hypotheses from the file. No paper reading, no new ideas. A counter in the hypothesis file enforces the boundary: after five engineering runs, the agent must stop and return to research, regardless of whether it wants to "just try one more thing."

This worked better. The agent produced five well-crafted hypotheses in its first research phase, including one that applied nGPT's learnable LERP residual connections asymmetrically (only to the MLP branch, not attention, because ReluSquared outputs are sparse directional vectors while attention outputs are dense superpositions) and another that applied DeepSeek-V2's multi-head latent attention compression to value embeddings along the vocabulary axis rather than the time axis. These were genuinely non-obvious syntheses that passed our falsifiability and novelty checks.

They also all failed experimentally. The nGPT-style per-dimension scaling conflicted with Muon's Newton-Schulz orthogonalisation. The low-rank VE compression lost per-token diversity that turned out to be load-bearing. But the failure analyses were precise and useful: each one identified exactly which part of the mechanistic story was wrong (the source primitive, the mapping, or the target bottleneck diagnosis), and these analyses fed into the next research phase.

## The Throughput Trap

By version three, we had tested about 17 paper-driven hypotheses across multiple runs, and none had produced a confirmed improvement that engineering alone would not have found. We began to suspect the benchmark itself was the problem, not the research methodology.

The culprit was the time budget. When training is bounded by wall-clock time, every architectural change is evaluated through a throughput lens. A wider MLP that processes 17% fewer steps in the same time budget appears to regress, even if each step is more informative. An auxiliary loss that adds gradient computation overhead appears to hurt, even if the gradient signal is richer. The agent had learned this and recorded it in findings: "throughput is king" and "any change that reduces step count must provide proportionally more quality per step."

The fix was almost embarrassingly simple. We replaced the time budget with a step budget. Instead of "train for 7.5 minutes," the instruction became "train for exactly 1800 optimiser steps." Now every architecture gets the same number of learning opportunities, regardless of how long each step takes. A wider MLP that is slower per step still gets 1800 steps. A model with an auxiliary loss still gets 1800 updates.

The results were immediate and striking. The 6x MLP expansion hypothesis, which had been conclusively "refuted" under time-bounded training (it lost 17% of steps), now worked. It improved val_bpb by 0.005. More dramatically, the accumulated finding that "ReluSquared is better than SwiGLU," which had been confirmed across 117 runs and recorded as established fact in the knowledge base, turned out to be false. SwiGLU is slower per step (because it uses all neurons, whereas ReluSquared creates roughly 95% sparsity), so under time-bounded evaluation it appeared worse. Under step-bounded evaluation, SwiGLU won by a substantial margin. The entire "confirmed mechanism" was a measurement artefact.

This was, to our knowledge, a genuinely novel methodological finding. Time-bounded benchmarks systematically favour architectures with sparse activations because sparsity correlates with throughput. Dense activations that use the full hidden dimension are penalised not because they learn less per step, but because they complete fewer steps. The standard practice of comparing architectures under fixed compute budgets (measured in wall-clock time or FLOPs) may produce false conclusions about architectural quality.

## The Clean Experiment

Armed with step-bounded training, we designed a final experiment to isolate the research contribution. All hyperparameters were frozen: learning rates, batch size, warmdown schedule, weight decay, softcap. All model dimensions were frozen: depth, width, head count, MLP expansion ratio. The agent could only modify the architecture itself, through changes grounded in cross-domain paper synthesis. No Tier B refinements. No "just adjust the learning rate for this new component." If the idea works at the default hyperparameters, it works. If not, the structural failure analysis tells you why.

This version produced three confirmed architectural improvements that stacked on top of each other. The first was a pre-attention causal depthwise convolution, synthesised from Conformer (speech recognition) and H3 (state space models). The second was per-head learnable attention temperature, adding only 50 parameters. The third, and most interesting, was partial RoPE: applying rotary positional embeddings to only 75% of the head dimension, leaving 25% position-free for pure content-based attention.

The partial RoPE hypothesis was derived from a cross-domain synthesis between positional encoding analysis and OFDM (orthogonal frequency-division multiplexing) from telecommunications. The reasoning was that full RoPE forces all query-key dot product dimensions to be position-modulated, preventing the model from computing pure content similarity. By analogy with OFDM's principle of dedicating orthogonal subcarriers to different information types, the agent proposed allocating 75% of head dimensions to position-aware matching (where to attend) and 25% to position-free semantic matching (what to attend to). This produced the largest single improvement in the entire experiment: a val_bpb reduction of 0.0035, which is substantial for a single activation-level change at frozen hyperparameters.

## What We Actually Learned

The honest assessment is nuanced. The interventions that the agent discovered are not novel. Pre-attention convolution is a known technique. Learnable attention temperature is standard. Partial RoPE has been explored (CodeLlama uses it for length generalisation). An ML engineer familiar with the literature might have proposed any of these.

What is novel is the reasoning path. No human has, to our knowledge, derived partial RoPE from OFDM subcarrier allocation theory. The telecommunications analogy provides a principled design framework: the 75/25 split emerges from channel capacity arguments, not from trial and error. Whether this constitutes "research" depends on your definition. If research means discovering mechanisms that no one has proposed in any form, the system has not done that. If research means arriving at known mechanisms through novel chains of reasoning that connect previously unconnected fields, the system has done exactly that.

The more significant finding is methodological. The five iterations of the control protocol (single loop, phase separation, hard counter enforcement, step-bounded evaluation, frozen-HP ablation) each addressed a specific failure mode revealed by the previous version. The progression tells a story about the structural barriers to creative AI reasoning: an agent will default to safe incrementalism unless the prompt architecture makes it structurally impossible. Even then, the evaluation methodology (time-bounded vs step-bounded) can create systematic biases that masquerade as confirmed findings. The "ReluSquared is better than SwiGLU" result survived 117 experiments across two hardware platforms before the change to step-bounded evaluation revealed it as an artefact.

This suggests that the challenge of making AI agents do genuine research is not primarily about the agent's reasoning capability. The hypotheses generated by the research phase were consistently creative, mechanistically grounded, and falsifiable. The challenge is in the experimental design: the evaluation methodology, the constraints that prevent safe-mode drift, and the separation of concerns between the part of the system that generates ideas and the part that tests them. These are human research design problems, not AI capability problems.

The system we ended up with, including the two-phase loop, the hard counter, the step-bounded evaluation, and the frozen-HP ablation, is probably more interesting as a contribution to the methodology of AI-assisted research than as a source of novel ML architectures. The finding that time-bounded benchmarks create systematic throughput biases is independently useful. And the observation that an LLM agent can arrive at known techniques through genuinely novel cross-domain reasoning, reading a telecommunications paper and a positional encoding paper and concluding that attention heads should allocate orthogonal channels to position and content, suggests that the path to AI-driven scientific discovery may run through better experimental design rather than more capable models.
