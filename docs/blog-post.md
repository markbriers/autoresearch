# Teaching an AI Agent to Read Papers: What Happens When You Give an Autonomous ML Engineer Access to the Scientific Literature

**TLDR.** I extended Karpathy's autoresearch benchmark with academic literature access and tried, across nine iterations, to make the agent behave like a researcher rather than an engineer. **Best research-only val_bpb improvement: 1.03% at frozen hyperparameters (v6, eight stacked interventions); individual cross-domain results up to 0.69% from syntheses such as wavelet shrinkage theory → learnable activation threshold (v8).** For context, [Paper Lantern](https://www.paperlantern.ai/blog/auto-research-case-study) reports 4.05% versus 3.67% on the same benchmark with and without literature lookup, but their setup lets the agent tune hyperparameters and scale the model freely; mine freezes both from v5 onwards to isolate the architectural research contribution.

For the first eight versions, the agent produced novel reasoning paths to known mechanisms but never a genuinely novel mechanism. In v9, after integrating Bayesian surprise (independent probability estimates from Researcher and Evaluator subagents, with belief divergence used to prioritise testing), the system produced a PD-controller residual scaling mechanism that two independent comprehensive prior art searches could not find in the published literature. Both agents predicted it would fail. The surprise framework identified it as the most informative result precisely because both agents' models were wrong.

Most of the work was in the harness, not the model. The deeper question this raises is whether coding agents need a fundamentally different post-training objective — one that rewards epistemic curiosity rather than task completion — to explore without being forced. The ingredients exist separately ([AutoDiscovery](https://arxiv.org/abs/2507.00310) uses Bayesian surprise for scientific exploration; [CD-RLHF](https://aclanthology.org/2025.acl-long.1146.pdf) applies curiosity rewards during post-training), but nobody has combined them for coding agents. Total cost: about £100. Code, results, and research logs are in the repository.

---

In early March 2026, Andrej Karpathy released [autoresearch](https://github.com/karpathy/autoresearch), an open-source benchmark that hands a coding agent (Claude Code, in his original setup) a small GPT training script and lets it optimise val_bpb autonomously. The agent has shell access via Bash, can edit files, and runs in a tight loop: read code, modify, train for a fixed wall-clock budget, measure bits-per-byte, keep improvements, revert failures. On a given machine you can run roughly twelve experiments per hour with no human in the loop.

Watching the agent work, I noticed something. Every improvement was an engineering optimisation: adjust the learning rate, reduce the batch size, tweak the warmdown. The agent behaved like a systematic junior engineer, not a researcher. It never consulted the literature. It never formed a hypothesis grounded in theory. It never made a surprising connection between fields. This is not a Claude-specific failure — it's how coding agents in general behave under task-completion framing.

This got me thinking. Could I augment the loop with access to the scientific literature, and if so, would the agent actually use it to do something that looked more like research?

I'm not the only person asking this. [Paper Lantern](https://www.paperlantern.ai/blog/auto-research-case-study), an MCP server giving agents access to two million CS papers, ran a controlled comparison on the same benchmark and reported a 4.05% improvement with literature access versus 3.67% without. Their agent treats the literature as a lookup table — searching for known techniques and applying them. I wanted to test something harder: whether the agent could synthesise ideas across unrelated fields and produce reasoning that a competent engineer would not have arrived at by staring at the code.

What follows is a case study in nine iterations. The main contribution is not a new architecture but a set of observations about how benchmark design, process constraints, and prompt-level framing determine whether an agent explores creatively or defaults to safe engineering.


## The First Few Iterations (v1–v3): How Hard the Defaults Pull

I extended the agent's tool access. I gave it an arxiv MCP server (search papers, download, read full text in markdown), a custom skill wrapping the OpenAlex API for relevance-ranked search across hundreds of millions of academic works, and a system prompt that demanded cross-domain synthesis before any code change. I added a three-tier persistent memory system: a research log capturing every run, a findings file distilling generalisable insights, and a long-term memory file promoted across sessions.

It mostly didn't matter. Across the first 117 experiments (40 on a MacBook, 77 on a RunPod H100), only three were paper-driven. The rest were the same hyperparameter tuning Karpathy's original system produces. The new tools and memory were available; the agent rarely chose to use them. Its expected-value calculation always favoured "use Bash to edit train.py and adjust the learning rate" over "use the arxiv MCP to read a paper from control theory and try to connect it to attention."

In v2 I separated the loop into two phases with a hard counter: Phase 1 reads papers and writes hypotheses, Phase 2 implements them, and after five engineering runs the agent must return to research. This produced better hypotheses, but they all failed experimentally. The failure analyses were precise and useful — but by v3 I had tested about 17 paper-driven hypotheses across multiple runs and none had produced a confirmed improvement that engineering alone wouldn't have found. The underlying problem was something else.


## The Throughput Trap (v4)

The culprit was the time budget. When training is bounded by wall-clock time, every architectural change is evaluated through a throughput lens. A wider MLP that processes 17% fewer steps in the same time appears to regress, even if each step is more informative. The fix was embarrassingly simple: replace the time budget with a step budget. Instead of "train for 7.5 minutes," "train for exactly 1800 optimiser steps."

The results were immediate and slightly humbling. The accumulated finding that "ReluSquared is better than SwiGLU," recorded as established fact across more than a hundred experiments, reversed. SwiGLU is slower per step (it uses all neurons; ReluSquared is ~95% sparse), so under wall-clock evaluation it always appeared worse. Under step-bounded evaluation it won by 0.004. Within this benchmark, switching from wall-clock to fixed-step evaluation reversed several earlier conclusions. This isn't a claim that fixed-compute evaluation is generally a mistake — it's standard in scaling-law work. But in a wall-clock-bounded loop where the agent can change both the architecture and the step count simultaneously, throughput can swamp per-step learning quality and reverse apparent architecture rankings.


## The First Clean Experiment (v5)

Armed with step-bounded training, I designed an experiment to isolate the research contribution. All hyperparameters were frozen — learning rates, batch size, warmdown, weight decay, softcap. All model dimensions were frozen — depth, width, head count, MLP expansion. The agent could only modify the architecture itself, through changes grounded in cross-domain paper synthesis. No follow-up tuning. If an idea worked at the default hyperparameters, it worked. If not, the failure analysis recorded why.

A "confirmed improvement" here means a change that reduced val_bpb relative to the frozen baseline in a single run. Seed variance is approximately 0.002 val_bpb (measured in one spot check), so improvements smaller than that should be treated as directional signals rather than statistically robust findings. Multi-seed validation would strengthen all these claims.

v5 produced three architectural changes that stacked: a pre-attention causal depthwise convolution (-0.0017), per-head learnable attention scaling (-0.0002), and partial RoPE (-0.0035). Cumulative reduction from baseline: 0.0054.

The partial RoPE result was the interesting one. The agent's reasoning paired positional encoding analysis with OFDM (orthogonal frequency-division multiplexing) from telecommunications. By analogy with OFDM's principle of dedicating orthogonal subcarriers to different information types, the agent proposed allocating 75% of head dimensions to position-aware matching and 25% to position-free content matching. Partial rotary allocation has clear prior art (GPT-NeoX's `rotary_pct`), but I'd never seen anyone derive it from telecommunications subcarrier theory. The mechanism is known. The reasoning path isn't.


## The Rediscovery Problem (v6 → v7)

v5's three confirmed improvements were all known techniques. Pre-attention convolution exists in Conformer and Primer. Learnable attention scaling is well-precedented. Partial RoPE is in GPT-NeoX. The system was finding things that worked, but finding them by retracing paths other researchers had already walked.

For v6 I added adversarial self-critique, quantified belief tracking, and stronger prior art search (drawing on Google's AI Co-Scientist and AI2's AutoDiscovery). v6 produced eight confirmed improvements stacking from 0.9595 to 0.9496. All eight were still known techniques. By v7 I was convinced the bottleneck was the single-session phase simulation: when one agent generates a hypothesis, implements it, and judges the result, it has a stake in every step. [Anthropic's harness design article](https://www.anthropic.com/engineering/harness-design-long-running-apps) makes exactly this point about coding agents grading their own work.

I split the protocol into three Claude Code subagents — Researcher, Implementer, Evaluator — each defined in its own `.claude/agents/*.md` file with a distinct system prompt and a restricted tool list. Each subagent runs in its own isolated context window. The Researcher has full tool access (Bash, Read, Edit, Write, WebFetch, WebSearch, plus the arxiv MCP). The Implementer has Bash and file editing tools but no web access — it cannot read papers or second-guess the design. The Evaluator has *only* Read and Edit; no Bash, no web, no MCP. It physically cannot run code, search, or do anything that creates attachment to outcomes. It can only read files and write verdicts against a hard threshold. I wrapped the whole thing in an external bash loop that starts a fresh Claude Code session per cycle, so there is zero shared context between cycles — the persistent memory lives entirely in the tracking files on disk.

v7 ran one cycle. Three of four hypotheses were confirmed. The problem: the cross-domain pairings were weak. The Researcher was pairing Huang 2024 with Shazeer 2020 (two activation-function papers) and calling it cross-domain. The "two distinct domains" instruction had a loophole — different ML subfields qualified.


## The One-Line Fix (v8)

v8 was the cheapest and most effective change of the entire project. I added one sentence to the Researcher's system prompt: each domain pairing must contain exactly one ML paper and one paper from outside machine learning, deep learning, and optimisation. Valid Domain B sources include signal processing, control theory, information theory, neuroscience, physics, biology, telecommunications, compressed sensing, statistical mechanics. The Evaluator gained a matching rejection rule in its own system prompt — non-compliant contracts get blocked at the approval gate before the Implementer ever sees them.

The effect was immediate. The Researcher's next batch paired Shazeer's GLU variants with Candes and Tao on compressed sensing. Ma's MEGA attention with Brown's 1963 exponential smoothing. Gemma-2's softcap with Carandini and Heeger on divisive normalisation in cortex. DeepNet's residual scaling with Mallat's multi-resolution wavelet analysis. Gemma-2's logit softcap with Jaynes's 1957 maximum entropy paper. The pairings were finally genuinely cross-domain.

Three hypotheses were confirmed before the pod ran out of funds. The cleanest was **ShrinkReLU**, derived from Donoho and Johnstone's 1994 wavelet shrinkage paper. The agent's reasoning chain: Donoho and Johnstone proved that soft thresholding is minimax-optimal for removing Gaussian noise from a signal, with the optimal threshold scaling with the noise level. Map this onto MLP activation design — small activations are likely noise from imperfect earlier layers; large activations are signal. Make the threshold learnable per layer so each layer adapts its denoising strength. The implementation: replace `F.relu(x).square()` with `F.relu(x - tau).square()` where tau is a softplus-parameterised learnable scalar. Ten extra parameters total. The result was a val_bpb reduction of 0.0058.

I think ShrinkReLU is the cleanest example of the "known technique via novel reasoning path" pattern across all nine versions. Soft thresholding is textbook signal processing. Learnable ReLU biases have been explored. But the specific combination, justified by wavelet shrinkage theory as activation denoising in a Muon-optimised transformer, doesn't appear in the literature in that form. The theoretical justification is load-bearing. An engineer trying "add a learnable bias to ReLU" faces an infinite design space — positive, negative, per-channel, per-layer, fixed, scheduled. Donoho and Johnstone tell you specifically: positive, per-layer, initialised small, learned, because optimal denoising thresholds scale with the noise level at each stage of processing. The theory collapses the design space to a single specific configuration with a principled reason for why it works.

The second confirmed result was a dimension-reduced SwiGLU, with the hidden-dimension choice justified by Candes and Tao's bounds on sparse signal recovery (sparse recovery needs only enough measurements to satisfy the recovery threshold, so 1728 rather than the full 4× expansion was sufficient). The third was a learnable softcap temperature derived from Jaynes's maximum entropy principle: the optimal entropy constraint should match the true entropy of the target distribution, which varies with training progress, so the softcap shouldn't be a fixed constant.

v8 also surfaced a problem I didn't get to solve. Stacking attempts on top of confirmed improvements often catastrophically regressed (ShrinkReLU plus learnable RMSNorm gain produced +0.136 val_bpb), apparently because both modify the same activation statistics. The Evaluator would have issued pattern directives about stacking risk in later cycles, but the pod ran out of funds before cycle 3.


## Bayesian Surprise and the First Novel Mechanism (v9)

For the final iteration I integrated the Bayesian surprise principle directly into the subagent protocols. The key addition was independent probability estimates: when the Researcher writes a sprint contract, it includes its own P(success) estimate. The Evaluator, which has not seen the Researcher's reasoning process (only the contract), independently writes its own P(success). The divergence between the two estimates is itself a signal: high divergence means the agents disagree about whether this will work, which indicates genuine uncertainty about the design space. High-uncertainty hypotheses are prioritised for testing because they have the highest expected information gain regardless of outcome.

After each run, the Evaluator computes a surprisal score based on how far the outcome was from both predictions. If both agents predicted success and the result was catastrophic: high surprisal. If both predicted failure and it succeeded: high surprisal. If predictions diverged and the outcome resolved the disagreement: medium. If the outcome matched both predictions: low. The Evaluator then issues exploration directives: high-surprisal subsystems get flagged for further investigation (we understand them least), low-surprisal subsystems get deprioritised (outcomes are predictable, diminishing returns).

v9's first cycle produced four hypotheses, all with genuine cross-domain pairings. H1 paired SwiGLU with Carandini and Heeger's divisive normalisation from neuroscience (INCONCLUSIVE at -0.0025, below the -0.003 threshold). H2 applied divisive normalisation directly to MLP activations (REFUTED at +0.0047). H3 used Heun's method from numerical analysis to build predictor-corrector residual connections (INCONCLUSIVE at -0.0006). And H4, which both agents gave the lowest probability of success, applied proportional-derivative control theory to the residual stream.

H4 deserves detailed description. The Researcher's reasoning was that classical PD controllers use two terms: a proportional term responding to the current state, and a derivative term responding to the rate of change. In a transformer, the standard residual connection is purely proportional: `x[l+1] = x[l] + f(x[l])`. Adding a derivative term would let the model anticipate where the residual stream is heading and correct accordingly. The implementation computes the "velocity" of the residual stream as `v[l] = x[l] - x[l-1]` (the difference between consecutive layer outputs), scales it by a learnable per-layer coefficient `deriv_lambda` (initialised at zero for safe passthrough), and adds it as a correction. Ten extra scalar parameters, negligible VRAM.

Both agents predicted it would fail. The Researcher gave P(success) = 0.20, writing in the case for failure that "the discrete derivative of a 10-step sequence is inherently noisy." The Evaluator independently estimated P(success) = 0.15. Belief divergence was just 0.05, the lowest in the batch, meaning both agents agreed it wouldn't work. The Evaluator's pre-run notes explicitly flagged it as having lower expected information gain than H3 due to the low divergence.

It produced the best result of the cycle: val_bpb reduction of 0.0042, clearly above the -0.003 threshold. The Evaluator scored it HIGH surprisal and wrote: "This reveals a gap in both agents' models: the derivative term in the residual stream provides meaningful signal even at depth=10, contrary to expectations. The discrete derivative is too noisy concern was wrong."

I then ran a thorough prior art search using both Claude and ChatGPT, structured around six dimensions: papers computing inter-layer differences as forward-pass signals, PID/PD control applied to transformer residuals, derivative/momentum terms in residual streams, Neural ODE methods beyond Euler/Heun, the specific PCformer paper, and surveys on transformers as dynamical systems. Both models converged on the same assessment.

The closest prior art is [Momentum Residual Neural Networks](https://arxiv.org/abs/2102.07870) (Sander et al., ICML 2021), which adds a velocity state to residual networks. But Momentum ResNets maintain a separate velocity through an exponentially weighted moving average of sublayer outputs (`v[n+1] = gamma * v[n] + (1-gamma) * f(x[n])`), using a fixed global gamma (typically 0.9). The H4 mechanism computes the raw inter-layer difference `x[l] - x[l-1]` and scales it by a learned per-layer coefficient initialised at zero. These are related but mechanistically distinct: one is an accumulated momentum of function outputs, the other is a measured trajectory derivative used as a closed-loop correction.

The second closest is [PIDformer](https://arxiv.org/abs/2402.15989) (ICML 2024), which genuinely applies PID control theory to transformers. But PIDformer's derivative term operates on error relative to a reference signal (the scaled initial embedding), not on the inter-layer finite difference of the residual stream. Different mechanism, different theoretical target.

Neither search found a published paper that computes `v[l] = x[l] - x[l-1]` during the forward pass and feeds it back as a learned per-layer correction to the transformer residual stream. The inter-layer difference is widely used as an analysis tool (tuned lens, logit lens, residual stream norm growth), but purely for interpretability, never as an architectural signal.

This means H4 is the first genuinely novel mechanism the system produced across nine versions. Not a novel reasoning path to a known mechanism (like partial RoPE via OFDM or ShrinkReLU via wavelet shrinkage), but a mechanism that, as far as two independent comprehensive searches can determine, has not been published in this form. The PD control framing from classical control theory, applied to the transformer residual stream with a learned derivative coefficient, appears to be a new contribution to the transformer architecture literature.

The irony is not lost on me. Both agents deprioritised H4 during planning. It had the lowest belief divergence (both agreed it would fail) and the Evaluator recommended testing other hypotheses first. The Bayesian surprise framework correctly identified it after the fact as the highest-information-gain result, but only because the agents' models were wrong in exactly the way that makes surprise informative. The system found something genuinely new not because it was looking for it, but because it tested something neither agent believed in.

The system continued running for six cycles and 17 experiments. Three more results cleared the hard threshold. H6 (SwiGLU with a wider hidden dimension, -0.0050) was a known technique, but the Evaluator had authorised it as a follow-up after H1's narrower SwiGLU was INCONCLUSIVE. H14 (factored embeddings with a 256-dimension bottleneck) was the second high-surprisal finding: both agents gave it P(success) below 0.15, and the Researcher explicitly assigned P(regression) = 0.25, yet it confirmed at -0.0034. The Evaluator's analysis: "the embedding layer is over-parameterised at this scale and factored embeddings with Muon-optimised projection are a productive direction — a finding neither agent anticipated." H22 confirmed that PD residual scaling and factored embeddings compose well (-0.0034 as a stack).

The stacking experiments were the most revealing part of the extended run. The system tested what happens when you combine confirmed improvements, and the answer is: it depends. SwiGLU plus PD residuals (H10) produced -0.0057, close to the sum of the individual deltas. But adding factored embeddings on top (H15, the triple stack) actually regressed to -0.0031, worse than the double. SwiGLU plus factored embeddings without PD (H19) was also worse than SwiGLU alone. The Evaluator caught this and scored it HIGH surprisal: "The assumption that confirmed improvements compose monotonically is violated." It characterised the interaction as a 37% subadditive discount and noted that SwiGLU and factored embeddings specifically interfere, while PD residuals compose well with either.

The per-head attention temperature result (H8) was the second-most-informative negative result in the programme. Both agents predicted neutral-to-mild improvement; the result was a clear regression of +0.0046. The Evaluator's analysis: "the attention subsystem actively resists scalar modifications even when initialised at identity. The mechanism is QK-norm plus softcap creating a rigid logit regime." This is a transferable finding about the interaction between QK-normalisation and learnable attention scaling that contradicts v5 and v6's earlier results (where per-head temperature was confirmed under a softer threshold). The hard -0.003 threshold and the separated Evaluator produced a different, and arguably more honest, assessment.

The Evaluator's cycle-by-cycle surprisal summaries are worth reading in full (they are in the repository at `h100-results-v9/evaluations.md`). They show the system learning over time: the cycle 1 summary identifies PD residual scaling as the key unexpected finding, the cycle 2 summary identifies the attention rigidity pattern, cycle 3 identifies embedding over-parameterisation and subadditive stacking, and by cycles 4-5 the system is correctly predicting most outcomes (low surprisal, few model updates needed). The Bayesian surprise framework produced diminishing returns as the agents' models calibrated, which is exactly what should happen: a well-calibrated agent makes fewer surprising predictions, and the information gain per experiment decreases. This is a natural stopping criterion that the system was approaching by cycle 6.

The concern about compounding results making further innovation harder proved well-founded. Each confirmed improvement changes the baseline, and the stacking experiments showed that improvements can interfere destructively. The design space narrows as the model accumulates modifications. The system's response to this narrowing was also informative: by cycle 5, the Evaluator was issuing LOW SURPRISAL ratings on most hypotheses ("outcomes are predictable, diminishing returns"), which suggests the agents had learned the productive regions of the design space and were running out of new territory to explore at this model scale.

## What I Actually Learned

**Across v1-v8, none of the interventions were novel mechanisms.** Pre-attention convolution, learnable scaling, partial RoPE, ShrinkReLU, SwiGLU, learnable softcap — all have clear precedents. The system was producing novel reasoning paths to known mechanisms, and only when the harness forced it to.

**In v9, the system produced what appears to be a genuinely novel mechanism.** PD-controller residual scaling — computing the inter-layer velocity of the residual stream and applying a learned derivative correction — has no direct published precedent that two independent comprehensive searches could find. The closest prior art is [Momentum Residual Neural Networks](https://arxiv.org/abs/2102.07870) (Sander et al., ICML 2021), which maintains a separate velocity state through EMA of sublayer outputs with a fixed global coefficient, and [PIDformer](https://arxiv.org/abs/2402.15989) (ICML 2024), which applies PID control to error relative to a reference signal, not to inter-layer differences. Both agents predicted this hypothesis would fail. The Bayesian surprise framework identified it as the most informative outcome precisely because both agents' models were wrong. The system found something new not by looking for novelty, but by testing what neither agent believed in.

**The Bayesian surprise framework also produced two other high-surprisal findings.** Factored embeddings (H14) were predicted to fail by both agents (P below 0.15) and confirmed, revealing that the embedding layer is over-parameterised at this scale. Per-head attention temperature (H8) was predicted to help and caused a clear regression, revealing that QK-norm plus softcap creates a rigid logit regime that resists scalar modifications. And the stacking experiments showed that confirmed improvements interact subadditively at a 37% discount, with some pairs (SwiGLU and factored embeddings) actively interfering. These are transferable architectural findings that emerged because the system tested hypotheses that both agents rated as low-probability.

**Whether this constitutes "research" is now a cleaner question.** In v5-v8 the answer depended on definition: novel reasoning paths to known mechanisms, maybe research, maybe not. In v9 the mechanism itself appears novel (pending more thorough literature review), and the framework that produced it (independent probability estimates, belief divergence, surprisal-weighted exploration) is well-characterised and reproducible. A single confirmed novel result across 200+ experiments and nine protocol iterations is not a high hit rate. But it is a non-zero hit rate, and the system was approaching a natural stopping criterion: by cycle 6, the Evaluator was rating most outcomes as LOW surprisal, meaning the agents' models had calibrated and the information gain per experiment was diminishing.

**The agent will default to engineering unless every loophole is closed.** A single-loop agent does Tier B tuning. Phase separation gets circumvented. A hard counter gets relabelled. A two-domain rule gets interpreted as different ML subfields. The load-bearing element was always the constraint that was hardest for the agent to reinterpret creatively. A sceptical reader might also note that 200+ adaptive experiments against a single validation metric look a lot like benchmark gaming. They'd be right — the frozen-HP constraint in v5+ prevents the most obvious form, but architectural changes are still selected against the same metric. Multi-seed validation would strengthen all of this; I didn't do it.

**The deeper issue is exploration-exploitation, not autoresearch specifically.** Colleagues working with coding agents (Cursor, Devin, Codex, Claude Code) in other contexts report the same dynamic: the agent settles into execution mode (tight feedback, clear next steps, reliable small gains) and resists switching to research mode (ambiguous goals, deferred reward, higher variance). Coding agents are post-trained to be helpful task-completing systems — rewarded for solving the problem in front of them, producing working code, converging on an answer. That training objective directly selects for exploitation. The research mode I wanted (read broadly, form speculative hypotheses, tolerate ambiguity, value information gain from failed experiments) is essentially the opposite of what the post-training optimises for.

The ingredients for a fix exist separately in recent work. [AI2's AutoDiscovery](https://arxiv.org/abs/2507.00310) uses Bayesian surprise as a reward function within Monte Carlo tree search to guide scientific exploration at inference time: the LLM's prior and posterior beliefs about hypotheses are extracted via sampling, and surprisal quantifies the information gain. Under fixed budget, it produces 5-29% more discoveries deemed surprising by both the LLM and domain experts. [Curiosity-Driven RLHF (CD-RLHF)](https://aclanthology.org/2025.acl-long.1146.pdf), published at ACL 2025, applies curiosity as an intrinsic reward during post-training itself, reporting 8-40% diversity gains while maintaining alignment. AutoDiscovery proves that Bayesian surprise works as an exploration signal for LLM agents doing science. CD-RLHF proves that curiosity-driven rewards can be applied during post-training without destroying alignment. But nobody has combined the two: using Bayesian surprise or information gain as a post-training objective specifically to make coding agents more exploratory in research contexts. A model post-trained this way would weight a calibrated prediction that fails interestingly higher than a safe prediction that succeeds boringly. Whether that would reduce the load on the harness — whether the agent would explore *without* being forced — is the question I find most interesting coming out of this work.

**Single-line summary:** Across nine versions and 200+ experiments, the system produced one apparently novel mechanism (PD residual scaling from control theory), several novel reasoning paths to known mechanisms (partial RoPE via OFDM, ShrinkReLU via wavelet shrinkage, learnable softcap via maximum entropy), a set of transferable architectural findings (subadditive stacking, attention rigidity under QK-norm, embedding over-parameterisation), and a large body of well-documented negative results. Most of the work was in the harness: the system prompts, the tool restrictions, the context isolation between subagents, the persistent memory format, the cross-domain enforcement, and the Bayesian surprise integration that ultimately surfaced the results neither agent believed in. The system was approaching a natural stopping criterion by its final cycles, with the agents' models calibrating and the surprisal per experiment diminishing. Whether a fresh run with a different model scale, a different benchmark task, or a post-training objective that rewards epistemic curiosity would produce a higher novelty rate is the open question I find most worth pursuing next.
